The PostgreSQL runtime layer for daslang's SQL machinery. A connection is a
``SqlRunner`` — open one with ``with_postgres(conninfo) $(db) { ... }`` (RAII;
``conninfo`` is a libpq connection string like ``"host=localhost dbname=mydb"``
or a ``postgres://`` URL) — and everything else takes it as the first argument:
raw ``exec`` / ``try_exec``, typed CRUD over ``[sql_table]`` structs
(``create_table`` / ``insert`` / ``update`` / ``delete_`` / ``check_schema``),
``with_transaction`` / ``try_transaction``, and the ``run_select`` family that
``_sql(...)`` chains compile into.

PostgreSQL-specific behavior worth knowing (versus the SQLite reference
provider): every statement's ``?`` placeholders are renumbered to libpq's
``$1, $2, ...`` in one quote-aware pass at execute time (user-facing SQL keeps
the portable ``?`` spelling; don't mix dollar-quoted ``$tag$...$tag$`` bodies
with ``?`` binds in raw SQL); parameters travel in text format with explicit
type OIDs, so ``$n`` type inference never fails server-side; ``create_table``
renders strict PostgreSQL types (``BIGINT`` / ``DOUBLE PRECISION`` / ``TEXT`` /
``BYTEA``) and declares a single integer primary key ``GENERATED BY DEFAULT AS
IDENTITY``; ``insert`` reports the new primary key via ``INSERT ... RETURNING``;
nested ``with_transaction`` scopes compose as ``SAVEPOINT`` / ``RELEASE`` (same
as the SQLite reference), with ``in_transaction`` reading the connection's live
``PQtransactionStatus``; ``_each_sql`` iterates a materialized result (libpq
materializes query results anyway). Client-side ``[sql_function]`` UDFs are not
available — PostgreSQL runs no client code; use server-side functions from raw
SQL instead.
